I. The Vision: Agent-Native Operations

An agent-native organization doesn't just use AI — it structures its operations around autonomous agents the way a traditional organization structures around people. Agents have roles, responsibilities, managers, and performance expectations. Humans set strategy, approve external actions, and handle relationships. Agents execute.

"The difference between chaos and orchestration comes down to understanding isolation boundaries." — Zen Van Riel, OpenClaw Multi-Agent Orchestration Guide

This isn't about replacing humans. It's about building a force multiplier — a team of agents that lets one person (you) operate with the throughput of a department. Intelligence gathering, content production, research synthesis, communications drafting, contract monitoring, hiring pipeline management — all running in parallel, 24/7, with human oversight at critical decision points.

II. Orchestration Patterns

Microsoft, Databricks, and the OpenClaw community have identified several proven patterns for multi-agent coordination. Each has trade-offs:

🔗 Sequential Pipeline

Agent A → Agent B → Agent C. Each processes the output of the previous. Good for: research → write → review flows.

⚡ Concurrent Fan-Out

Orchestrator sends tasks to multiple agents simultaneously, then aggregates. Good for: parallel research, multi-source scanning.

🎯 Supervisor / Hierarchical

Manager agent delegates, monitors, and synthesizes. Workers report back. Good for: complex projects with sub-tasks. Our primary pattern.

🤝 Handoff

Agents pass control to the most capable specialist. Good for: customer service, multi-domain expertise.

Our approach: We use a hierarchical supervisor pattern with a Chief of Staff orchestrator at the top, domain managers in the middle, and specialist workers at the bottom. This mirrors how Changeis itself is organized — and how federal agencies structure their teams.

V. Phase 1: Foundation (Current)

Now — Week 1-2

This is where we are today. A single powerful agent (VM2) doing everything — research, writing, publishing, monitoring, task management. It works, but it's sequential and bounded by session context.

Chief of Staff + Everything Else

🧠 VM2 (Main Agent)

claude-opus-4-6 · workspace: /data/.openclaw/workspace
  • Daily VEB production (12 sections, audio, PDF, graphics)
  • Research via web search, Reddit, YouTube transcripts
  • Email monitoring (himalaya/Gmail)
  • Todoist task management (VM2 label)
  • Content drafting (LinkedIn, Transformation Signal)
  • Hiring materials (job posts, referral programs)
  • SAM.gov monitoring
  • Heartbeat scans every 15 minutes

What's working: Everything. One agent can handle remarkable breadth.

What's limited: Sequential execution. Can't research while writing. Heartbeat scans compete with user conversations for context window. Token-heavy — Opus for every task, even simple ones.

VI. Phase 2: Specialist Workers

Next — Week 3-4

Spawn dedicated worker agents for bounded, repeatable tasks. These run on cheaper models (Haiku or GPT-4.1) and operate in their own sessions. VM2 remains the orchestrator.

Specialist Worker

🔍 Intel Scanner

claude-haiku-4-5 · cron: every 4 hours
  • Reads topics.json for search keywords
  • Runs web searches on 3-4 topics per cycle
  • Scans Reddit (r/ATC, r/govcon, r/fednews)
  • Writes findings to intel-queue.json
  • VM2 (Chief of Staff) reviews queue during heartbeat

  • Classifies: urgent / newsletter / actionable / noise
  • Extracts key content from newsletters
  • Writes summaries to email-digest.json
  • Flags urgent items for immediate VM2 notification

IX. OpenClaw Multi-Agent Implementation

Key Concepts

Agent Isolation

Each agent in OpenClaw is fully isolated: own workspace, own session store, own auth profiles. Agent A cannot see Agent B's memory or credentials. This is critical for security — the SAM.gov agent shouldn't access your personal email.

Bindings

Bindings route inbound messages to the correct agent. Example: Telegram messages from you go to VM2 (Chief of Staff). Webhook calls from SAM.gov go to the SAM Watcher. Cron triggers go to the Intel Scanner.

Sub-agents vs. Persistent Agents

TypeLifetimeUse Case
Sub-agent (sessions_spawn)One task, then diesAd-hoc research, one-off VEB section, specific analysis
Persistent agent (agents.list)Always runningIntel Scanner, Email Processor, SAM Watcher
Cron agentWakes on scheduleDaily digest, weekly synthesis, scheduled scans

Inter-Agent Communication

Agents communicate through shared files (JSON queues in a shared directory) or sessions_send (direct message to another agent's session). The Chief of Staff reads from all queues and synthesizes.

// Shared intelligence queue structure
/data/.openclaw/shared/
├── intel-queue.json      # Intel Scanner → Chief of Staff
├── email-digest.json     # Email Processor → Chief of Staff
├── sam-opportunities.json # SAM Watcher → Chief of Staff
├── content-queue.json    # Chief of Staff → Content Manager
└── approval-queue.json   # Any agent → Human (via Chief of Staff)

X. Cost & Model Strategy

The key to affordable multi-agent operation: use the cheapest model that can do the job. Only the Chief of Staff needs Opus. Most workers can run on Haiku or even GPT-4.1.

T2 ManagersClaude Sonnet 4.5$3 / $15Task decomposition, quality review, summarization
T3 WorkersClaude Haiku 4.5$0.80 / $4Scanning, classification, simple extraction
T3 WorkersGPT-4.1$2 / $8Alternative for tasks needing different strengths

Cost projection: A Phase 2 setup (VM2 on Opus + 3 workers on Haiku) running heartbeat + cron jobs would cost roughly $2-5/day in API calls. Phase 3 with managers would be $5-10/day. Phase 4 at full autonomy: $10-20/day. That's the cost of a coffee to run a department.

XI. Governance & Safety

The Approval Gate

No agent can take external action without human approval. This is the non-negotiable rule that makes the entire system safe.

Action TypeApproval Required?Example
Internal read/writeNoSearch web, write to workspace, update memory
Internal analysisNoClassify email, score opportunity, draft content
Publish contentYes — human approvesLinkedIn post, tweet, newsletter send
Send emailYes — human approvesClient communication, outreach
Submit proposalYes — human approvesSAM.gov response, RFP submission
Financial actionYes — human approvesPurchase, subscription, contract

Monitoring & Observability

Mission Control (TenacitOS dashboard) provides visibility into all agent activity: which agents ran, what they did, how many tokens they consumed, and what they produced. The Activity Feed is your audit trail.

Guardrails

  • Token budgets: Each agent has a per-day token limit. Prevents runaway costs.
  • Timeout limits: Worker agents timeout after 10 minutes. Managers after 30.
  • Depth limits: Sub-agents cannot spawn sub-sub-agents beyond depth 2.
  • File isolation: Workers can only write to their own workspace + shared queue directory.
  • No credential sharing: Each agent has only the API keys it needs for its specific function.
